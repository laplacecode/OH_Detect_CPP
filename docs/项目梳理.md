# OH_Detect_Station_Module 项目梳理

本文档用于快速理解当前仓库的交付边界、验证链路和维护重点。正式交付内容仍以 `delivery/package/` 及根目录原始头文件为准。

本次文档整理采用保守范围：只更新 README 类说明文档和本文档，不改写原始接口说明 PDF/TXT、历史计划文档、教学文档、YOLO 标签文件或模型文件。

## 1. 项目定位

本仓库是 OH 检测分割算法的交付与验证仓库，不是完整业务应用。

当前内容分成三类：

- 正式交付物：`delivery/package/`
- 客户侧参考源码：`delivery/reference_qt_adapter/`
- 内部训练、导出和验证工具：`python_verify/`、`cpp_verify/`、`runs/`

核心接口形态来自原始头文件：

```text
initParas(QByteArray&)
reciveData(std::vector<cv::Mat>&, QByteArray&)
run()
sendData(std::vector<cv::Mat>&, QByteArray&)
```

协议字段是固定顺序 UTF-8 `QByteArray`，不是 JSON。

## 2. 交付边界

正式交付包：

```text
delivery/package/
  base.h
  OH_Detect_Station_Module.h
  doc/
  model/
    best.onnx
  README.md
```

边界约束：

- `delivery/package/` 只放客户正式接收的接口、文档和模型。
- `delivery/reference_qt_adapter/` 是参考源码附件，不替代正式交付包。
- `python_verify/`、`cpp_verify/`、`runs/` 不属于正式交付内容。
- `best.onnx` 不能单独代表完整模块，必须配套预处理、后处理和协议组包逻辑。

## 4. 运行主流程

外部集成调用顺序：

```text
initParas
  解码 36 个初始化字段
  解析模型路径
  初始化 ONNX Runtime 会话

reciveData
  解码 6 个单次触发字段
  接收至少 1 张 CV_16U 灰度图
  尺寸标准化、窗宽窗位转 8 位、图像增强

run
  中心裁切 768 x 768
  YOLOv8-seg ONNX 推理
  解析检测输出和 mask proto
  NMS、mask 重建、结果恢复到整图
  生成 sendData 中间字段

sendData
  输出 5 张图
  编码 23 个固定顺序结果字段
```

输出图顺序：

```text
1. m_drawingImage_8bit
2. m_rawImage_8bit
3. m_enhancedImage_8bit
4. m_segImage
5. m_segImageTo255
```

## 5. 模型状态

当前正式模型：

```text
delivery/package/model/best.onnx
```

已检查的 ONNX IO：

```text
input:
  images  tensor(float) [1, 3, 768, 768]

outputs:
  output0 tensor(float) [1, 37, 12096]
  output1 tensor(float) [1, 32, 192, 192]
```

这符合标准 YOLOv8-seg 双输出形态：一路检测候选框和 mask 系数，一路 mask proto。

## 9. 已知风险和维护点

1. `python_verify/test_Pit/data.yaml` 中的 `path` 仍指向旧的 `internal_verify/test_Pit`，当前仓库实际目录是 `python_verify/test_Pit`。后续训练前应统一该路径。
2. `runs/segment/test_Pit/args.yaml` 也记录了旧路径，这是历史训练记录，不应直接当成当前配置来源。
3. `delivery/package/` 应保持稳定，验证逻辑和实验代码不要放进去。
4. C++ 参考实现中的测量结果是当前验证链路里的代理几何值，不应宣称等价于原始生产 OH 算法。
5. 协议字段顺序是兼容关键点，新增字段或重排字段都会破坏客户侧对接。
6. ONNX 当前是固定输入尺寸 `[1, 3, 768, 768]`，改成动态尺寸或不同输出结构时，Python 和 C++ 后处理都需要同步调整。
7. `delivery/package/doc/` 下的 PDF/TXT 和根目录接口说明 TXT 属于原始接口说明基线，日常说明更新应优先写入 README 或 `docs/`。

## 10. 推荐维护顺序

短期：

- 修正 `python_verify/test_Pit/data.yaml` 的数据集根路径。
- 重新跑一次 `check_onnx_io.py` 和 `run_pdf_protocol_verify.py --promote-8bit`。
- 如 C++ 环境可用，再跑 `cpp_verify\build_mingw_manual_qt6.cmd` 和 smoke 程序。

中期：

- 为协议字段生成一份单独字段表，避免 Python、C++、PDF 文档之间靠人工同步。
- 增加一个轻量对齐检查：比较 Python 和 C++ 的 `initParas/reciveData/sendData` 字段数量和顺序。
- 把训练、导出、覆盖交付模型的流程写成明确的发布检查清单。

长期：

- 区分“交付模型版本”和“实验训练版本”，避免 `runs/` 里的临时产物被误认为正式交付。
- 为 `delivery/reference_qt_adapter/` 增加最小自动化验证，覆盖非法调用顺序、协议字段数量错误、非 `CV_16U` 输入等场景。
