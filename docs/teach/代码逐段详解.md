# `delivery/reference_qt_adapter/` 新手代码详解

本文档的目标不是只告诉你“这段代码做了什么”，而是尽量把它讲到你以后自己能顺着代码读下去。

你可以把这套代码理解成一句话：

`它把一张 16 位灰度图 + 一段 QByteArray 参数，经过预处理、ONNX 推理、后处理，再重新打包成客户接口需要的 5 张图和 sendData 字节流。`

---

## 1. 先建立整体脑图

这套参考适配器一共分成 4 层：

1. `protocol_codec.*`
   负责“拆字节”和“拼字节”。
   也就是把 `QByteArray` 变成结构体，或者反过来把结构体变成 `QByteArray`。

2. `image_pipeline.*`
   负责“图像预处理”。
   包括尺寸统一、16 位转 8 位、增强、裁切、把分割结果贴回整图。

3. `seg_runtime.*`
   负责“ONNX Runtime 推理 + 标准 YOLOv8-seg 双输出解析”。

4. `OH_Detect_Station_Module.cpp`
   负责把前 3 层串起来，形成你最终调用的接口：
   `initParas -> reciveData -> run -> sendData`

---

## 2. 先补最基础的 C++ 读法

如果你完全不会 C++，先记这几个规则：

- `#include`
  类似 Python 的 `import`，把别的头文件内容引进来。

- `.h`
  一般放“声明”，告诉编译器“有这些结构体、类、函数”。

- `.cpp`
  一般放“实现”，真正写函数怎么做。

- `struct`
  就是一组打包在一起的数据。
  你可以先把它理解成 Python 里的 `dataclass`。

- `class`
  和 `struct` 很像，但更常用于“数据 + 方法”的组合。

- `namespace`
  相当于名字空间，避免重名。
  你可以粗略理解成 Python 里的模块名。

- `const`
  表示“这个值不应该被修改”。

- `&`
  这里多数时候表示“引用”。
  你可以先理解为“直接操作原对象，不复制一份”。

- `*`
  这里有两种常见用法：
  - 声明指针
  - 解引用指针

- `std::unique_ptr<T>`
  表示“这个对象由这里独占管理，离开作用域后自动释放”。
  你可以先把它理解成“自动托管的指针”。

- `::`
  表示“某个命名空间或类下面的成员”。
  例如 `cv::Mat`、`QString::fromUtf8`。

## 3. 这套代码的调用顺序

你以后读这套代码，脑中一直记这个顺序：

1. `initParas`
   解码初始化参数，加载模型
2. `reciveData`
   解码单次输入参数，处理输入图像
3. `run`
   做 ONNX 推理，生成分割结果和 sendData 中间结果
4. `sendData`
   输出 5 张图和最终字节流

也就是说：

- `initParas` 是“开机”
- `reciveData` 是“喂一张图”
- `run` 是“真正干活”
- `sendData` 是“把结果拿出来”

## 4. `protocol_codec.h` 逐段解释

文件位置：
[protocol_codec.h](D:/code/OH_Detect_Station_Module/delivery/reference_qt_adapter/protocol_codec.h:1)

### 第 1-5 行

```cpp
#ifndef ...
#define ...
#include <QByteArray>
#include <QString>
```

含义：

- 前两行叫“头文件保护”。
  作用是避免这个头文件被重复包含多次。
- `QByteArray` 是 Qt 的字节数组类型。
- `QString` 是 Qt 的字符串类型。

### 第 7 行

```cpp
namespace reference_qt_adapter {
```

含义：

- 表示后面的名字都放进 `reference_qt_adapter` 这个命名空间里。
- 好处是不会和别的项目里的 `InitParasData`、`SegRuntime` 撞名。

### 第 9-47 行：`InitParasData`

这是一个结构体，专门承接 `initParas(QByteArray&)` 解码后的结果。

你可以把它理解成：

- 输入原本是一串字节
- 这串字节解码后，变成一个“参数对象”
- 这个对象里每个字段都对应 PDF 协议里的一个位置

比如：

- `onnx_model_path`
  模型路径
- `window_width` / `window_level`
  窗宽窗位
- `horizontal_oh_max_limit`
  某个阈值
- `enable_horizontal_oh`
  某个功能开关

为什么很多字段后面都有默认值？

- 因为结构体刚创建时，总要有个初始状态
- 这样即使某些字段暂时还没被赋值，也不会是完全随机值

### 第 49-57 行：`ReceiveDataData`

这是 `reciveData(QByteArray&)` 解码后的结果。

它表示“一次检测任务”的附加信息，比如：

- 相机名
- 产品二维码
- 管电压
- 管电流
- 治具号

### 第 59-84 行：`SendDataData`

这是最终 `sendData` 要输出的数据结构。

也就是：

- 先在程序内部计算出这些字段
- 最后再编码回 `QByteArray`

这里你可以把它理解成“sendData 的草稿纸”。

### 第 86-91 行：函数声明

```cpp
bool decode_init_paras(...)
bool decode_receive_data(...)
QByteArray encode_send_data(...)
```

含义：

- `decode_*`
  把字节流解码成结构体
- `encode_*`
  把结构体编码成字节流

返回 `bool` 的函数一般表示：

- `true` 成功
- `false` 失败

## 5. `protocol_codec.cpp` 逐段解释

文件位置：
[protocol_codec.cpp](D:/code/OH_Detect_Station_Module/delivery/reference_qt_adapter/protocol_codec.cpp:1)

这个文件真正实现了“字节 <-> 结构体”的转换。

### 第 8-18 行：`decode_lines`

这段代码做的事很简单：

1. 把 `QByteArray` 当成 UTF-8 文本读取
2. 按换行符 `\n` 分割成多行
3. 如果某一行末尾带 Windows 的 `\r`，就去掉

为什么这么做？

因为当前参考实现沿用了 Python 验证链路里的约定：

`一个字段一行`

也就是说，这里不是二进制结构体序列化，不是 JSON，而是：

```text
字段1
字段2
字段3
...
```

### 第 20-32 行：`parse_bool_token`

这段是布尔值解析器。

它的作用是把这些文本：

- `true`
- `1`
- `yes`
- `on`

都当成真。

把这些文本：

- `false`
- `0`
- `no`
- `off`

都当成假。

为什么要写这个函数？

因为协议里的布尔值不是 C++ 的 `true/false` 变量，而是字节流里的字符串。

### 第 34-50 行：`parse_int_token` 和 `parse_double_token`

这两个函数负责把字符串变成：

- `int`
- `double`

比如：

- `"123"` -> `123`
- `"90.0"` -> `90.0`

### 第 52-54 行：`append_bool`

这段代码目前没有被后面的 `encode_send_data` 用到。

它本来是一个辅助函数，用于把布尔值重新写成：

- `true`
- `false`

它不影响当前逻辑，你可以把它看成“预留的工具函数”。

### 第 56-64 行：`expect_count`

这段很重要。

它检查：

- 这次拆出来的字段个数
- 是否等于协议要求的字段个数

例如：

- `initParas` 应该有 36 个字段
- 如果只有 35 个或者 37 个，就说明数据坏了

这时候函数会返回 `false`，并写错误信息。

### 第 66-75 行：模板函数 `parse_with`

这是一个“通用解析助手”。

你可以先不要被 `template` 吓到。

它的核心想法很简单：

- 给它一个解析函数
- 给它一组 tokens
- 给它第几个字段
- 给它目标变量

它就尝试把该字段解析进去。

如果失败，就顺便把错误信息写清楚。

为什么要这样写？

因为：

- 解析 `double`
- 解析 `int`
- 解析 `bool`

这三类代码的流程很像，没必要每次重复写错误处理。

### 第 79-123 行：`decode_init_paras`

这是整个文件里最关键的函数之一。

它的流程是：

1. `decode_lines(raw)`
   先把字节流拆成多行
2. `expect_count(tokens, 36, error)`
   确认字段数必须等于 36
3. 前 3 个字段直接按字符串赋值
4. 后面的字段逐个用 `parse_with(...)` 转成 `double/int/bool`

这段代码为什么这么长？

因为它是在明确表达：

`第 3 个字段就是 pixel_to_physical_scale`
`第 4 个字段就是 window_width`
`第 5 个字段就是 window_level`

这是一种“显式协议映射”。

优点是：

- 非常清楚
- 不容易把字段顺序弄乱

缺点是：

- 看起来长

但对于工业协议代码，这种写法反而是安全的。

### 第 125-138 行：`decode_receive_data`

和上面逻辑一样，只是字段数更少。

这里要求：

- 一共 6 个字段
- 前两个是字符串
- 后四个是 `double`

### 第 140-167 行：`encode_send_data`

这段逻辑是“反方向”：

- 把 `SendDataData` 结构体里的字段
- 严格按协议顺序写到 `QStringList`
- 最后用 `\n` 拼接
- 再转回 UTF-8 字节流

为什么 `double` 要用：

```cpp
QString::number(value, 'g', 16)
```

意思是：

- 用比较通用的浮点格式输出
- 保留较高精度

这样做能减少浮点数被字符串化时的信息损失。

## 6. `image_pipeline.h` 逐段解释

文件位置：
[image_pipeline.h](D:/code/OH_Detect_Station_Module/delivery/reference_qt_adapter/image_pipeline.h:1)

这个文件是“图像预处理层”的对外声明。

### 第 9-14 行：`PreprocessedImages`

这是一个中间结果结构体，保存 3 张图：

- `standardized16`
  尺寸统一后的 16 位图
- `raw8`
  由 16 位经过窗宽窗位转换得到的 8 位图

如果你不记住裁切位置，就没法恢复回去。

一共 4 个主要函数：

  负责把局部结果贴回整图
- `build_drawing_image`

## 7. `image_pipeline.cpp` 逐段解释

文件位置：
[image_pipeline.cpp](D:/code/OH_Detect_Station_Module/delivery/reference_qt_adapter/image_pipeline.cpp:1)

### 第 8-18 行：`standardize_size`

这段代码做的是“尺寸统一”。

目标尺寸被设成：

```cpp
cv::Size target_size(1536, 1152);
```

逻辑是：

- 如果原图本来就是 `1536x1152` 或 `2048x1152`
  直接复制
- 否则
  强制缩放到 `1536x1152`

为什么这样做？

因为当前参考实现想先把输入整理到一个稳定范围，后面的裁切和恢复逻辑才容易统一。

### 第 20-34 行：`window_to_8bit`

这是医学/工业图像里常见的窗宽窗位处理。

原图是 16 位灰度：

- 像素值范围大
- 直接显示和推理不方便

于是这里做 4 步：

1. 计算上下界
2. 把像素值裁剪到这个范围
3. 归一化到 0~1
4. 再映射到 0~255

最后得到 `CV_8U` 的 8 位图。

### 第 36-44 行：`enhance_8bit`

这里用了两个增强步骤：

1. `CLAHE`
   局部对比度增强
2. `GaussianBlur`
   高斯模糊，适当平滑噪声

你可以先把它理解成：

- 先让图像细节更明显
- 再做一点平滑，避免太噪

### 第 48-72 行：`prepare_input_images`

这是预处理总入口。

先做输入检查：

- 图像不能是空的
- 图像必须是 `CV_16UC1`

然后依次调用：

1. `standardize_size`
2. `window_to_8bit`
3. `enhance_8bit`

最后把结果放到 `PreprocessedImages` 结构体中。

### 第 74-90 行：`build_model_input_bgr`

这里做的是“给 ONNX 模型准备输入图”。

流程是：

1. 先算出裁切大小 `crop_size`
2. 用中心裁切取出图像中间一块
3. 如果裁出来不是 `768x768`
   就再缩放成 `768x768`
4. 把灰度图转成三通道 BGR

为什么要转三通道？

因为当前 ONNX 输入是：

`[1, 3, 768, 768]`

也就是 3 通道输入。

### 第 92-101 行：`restore_crop_to_full_size`

这个函数是“贴回去”。

做法是：

1. 先创建一张全黑整图
2. 如果裁切时做过缩放，就把分割结果缩回原裁切尺寸
3. 再把这块结果拷贝回整图对应位置

你可以理解成：

- 模型只看了局部
- 但最后结果图要放回全图坐标系

### 第 103-111 行：`build_drawing_image`

这段代码做一个可视化图：

1. 先把增强后的灰度图转成彩色图
2. 从 `seg255` 提取轮廓
3. 用绿色画轮廓

作用是：

- 人眼更容易看出模型分割到了哪里

## 9. `seg_runtime.cpp` 逐段解释

文件位置：
[seg_runtime.cpp](D:/code/OH_Detect_Station_Module/delivery/reference_qt_adapter/seg_runtime.cpp:1)

### 第 10-14 行：`sigmoid`

YOLOv8-seg 的 mask 恢复要用 sigmoid。

这里还做了一个裁剪：

```cpp
std::clamp(x, -50.0f, 50.0f)
```

原因是：

- 指数函数 `exp` 在数值太大或太小时不稳定
- 先截断可以避免数值爆炸

### 第 16-18 行：`get_shape`

这是一个小工具函数，用来拿 ONNX 张量形状。

比如它可以拿到：

- `[1, 37, 12096]`
- `[1, 32, 192, 192]`

### 第 22-26 行：构造函数和析构函数

构造函数：

```cpp
SegRuntime::SegRuntime()
    : env_(ORT_LOGGING_LEVEL_WARNING, "qt_reference_adapter")
```

含义：

- 一创建 `SegRuntime`
- 就顺手创建 ONNX Runtime 的环境对象

析构函数 `= default` 表示：

- 用编译器默认行为即可

### 第 28-53 行：`init`

这是模型加载函数。

核心步骤：

1. 创建 `SessionOptions`
2. 设置图优化级别
3. 用模型路径创建 `Session`
4. 读取输入名字
5. 读取所有输出名字

为什么要把输入输出名字缓存下来？

因为后面调用 `Run(...)` 时需要这些名字。

如果中间任何步骤失败，就进 `catch`，把错误信息写到 `error`，并返回 `false`。

### 第 55-57 行：`is_ready`

很简单：

- 如果 `session_` 不为空，说明模型已经加载好了

### 第 59-83 行：`select_output_indices`

这个函数解决的问题是：

`虽然我们知道模型有两个输出，但不想硬编码“第 0 个一定是 boxes，第 1 个一定是 proto”。`

所以这里按“维度数”判断：

- 3 维输出 -> boxes
- 4 维输出 -> proto

为什么这样做更稳？

因为标准 YOLOv8-seg 的双输出就长这样。

### 第 85-159 行：`parse_detections`

这是整个后处理里最难的一段之一。

它在做的事情是：

1. 读取 boxes 张量和 proto 张量形状
2. 算出 proto 通道数
3. 判断 boxes 是哪种排布方式
4. 算出类别数
5. 遍历每一个候选 prediction
6. 找得分最高的类别
7. 低于阈值就丢掉
8. 读取框坐标
9. 读取 mask 系数
10. 组装成 `Detection`

你可以先不深究 YOLOv8 每一维的数学意义，先抓住结果：

`它把模型原始输出，变成了程序容易理解的一组候选目标。`

#### 第 101-105 行特别重要

```cpp
const int proto_channels = ...
const int class_count = num_channels - 4 - proto_channels;
```

这里在算：

- 前 4 个数是框
- 后面一部分是类别分数
- 再后面一部分是 mask 系数

所以：

`总通道数 - 4 - proto通道数 = 类别数`

### 第 161-187 行：`apply_nms`

NMS 的作用是去掉高度重叠的重复框。

流程：

1. 把 `Detection` 里的框和分数取出来
2. 调用 OpenCV 的 `cv::dnn::NMSBoxes`
3. 只保留被选中的目标

这样可以减少重复目标带来的干扰。

### 第 189-264 行：`build_segmentation_maps`

这是最核心的 mask 重建函数。

整体思路是：

1. 从 proto 张量里拿到底层 mask 特征
2. 对每个目标：
   - 用自己的 mask 系数和 proto 做线性组合
   - 得到一张实例 mask
   - 经过 sigmoid
   - 缩放到输入图大小
   - 只保留目标框范围内的区域
3. 再把所有实例 mask 汇总成：
   - `seg_map`
   - `seg_map255`

`seg_map` 是类别图：

- 背景 0
- 类别 1 -> 值 1
- 类别 2 -> 值 2

`seg_map255` 是可视化二值图：

- 有分割的地方就是 255
- 没有分割就是 0

#### 第 216-227 行为什么这么慢看起来又复杂？

因为它本质上在手工做：

`mask = coeff @ proto`

也就是把一个目标的 mask 系数和 proto 特征图组合成 mask。

这是从 YOLOv8-seg 原理直接翻译过来的实现。

### 第 266-354 行：`run`

这是 `SegRuntime` 对外最重要的方法。

它的流程是：

1. 检查 session 是否已初始化
2. 检查输入图是不是 `CV_8UC3`
3. 把图像写进 CHW 格式的浮点张量
4. 构造 ONNX 输入张量
5. 调用 `session_->Run(...)`
6. 读取输出形状
7. 找到 boxes/proto 输出
8. 解析 detections
9. 做 NMS
10. 重建 `seg_map` 和 `seg_map255`

#### 第 282-291 行为什么自己手工写张量？

因为 ONNX Runtime 要的是：

`[1, 3, H, W]`

而 OpenCV 默认图像内存布局更接近：

`H x W x C`

所以这里需要自己手工把图像搬到 CHW 结构里。

#### 第 312-318 行

这就是 ONNX 真正执行推理的地方。

```cpp
outputs = session_->Run(...)
```

你可以把它理解成：

`把输入喂给模型，然后拿回输出张量列表`

## 10. `OH_Detect_Station_Module.cpp` 逐段解释

文件位置：
[OH_Detect_Station_Module.cpp](D:/code/OH_Detect_Station_Module/delivery/reference_qt_adapter/OH_Detect_Station_Module.cpp:1)

这是整套参考实现的“总调度层”。

它直接实现了交付头文件里的接口。

### 第 1-15 行：头文件引用

```cpp
#include "../package/OH_Detect_Station_Module.h"
```

这一句非常关键。

它表示：

`这个 .cpp 不是自己重新定义一套接口，而是直接实现正式交付头文件里的接口。`

这正是为什么它能保持 Qt 接口形态一致。

### 第 20-25 行：`ClassGeometry`

这是一个辅助结构体。

它不是协议定义的一部分，而是内部计算时用的：

- 组件数量
- 质心 x 偏移
- 质心 y 偏移

### 第 27-40 行：`AdapterState`

这是整个文件最关键的数据结构之一。

它保存了当前对象运行时需要的状态：

- 初始化参数
- 单次输入参数
- 最终 sendData 字段
- ONNX Runtime 对象
- 预处理图像
- 裁切信息
- 分割图
- 当前是否已经 init / receive / run

为什么要单独做这个结构？

因为正式交付头文件里没有为这套参考实现预留这些私有成员。

所以作者用了“外置状态表”的方式来托管。

### 第 42-50 行：`state_map` 和 `state_for`

这两段是“实例状态管理器”。

思路是：

- 每个 `OH_Detect_Station_Module` 对象
- 在外部 map 里挂一份 `AdapterState`

这样就能在不改交付头文件的前提下，给每个对象维持一份独立运行状态。

### 第 52-59 行：小工具函数

- `to_wstring_path`
  把 Qt 路径转成宽字符串，方便 ONNX Runtime 用
- `compute_mean_16u`
  算平均灰度值

### 第 61-85 行：`compute_class_geometry`

这段代码会：

1. 从分割图里找出某个类别的所有像素
2. 计算连通区域个数
3. 计算这些像素的平均位置
4. 转成相对整图中心的偏移量

这就是后面“对齐度”“层数”等代理测量值的基础。

### 第 87-96 行：`evaluate_layer_counts`

这是层数子结果的判断逻辑。

如果没有开启层数检测，就直接返回 OK。

如果开启了，就比较：

- 正极层数是否达到目标
- 负极层数是否达到目标

两个都满足才返回 OK。

### 第 98-127 行：`evaluate_oh_proxy`

这段是 OH 子结果的代理判定逻辑。

注意关键词：

`proxy`

意思就是：

`它不是原始生产算法的真实 OH 算法，而是当前验证链路里的近似代理测量。`

它做的事情是：

1. 从分割结果里找到最靠左和最靠上的非零点
2. 乘上像素物理尺度
3. 加上补偿值
4. 和上下限比较

### 第 129-164 行：`evaluate_alignment_proxy`

和上一个类似，这里是对齐度代理判定。

逻辑是：

- 如果该检测项没开启，就不参与判断
- 如果开启了，就拿质心偏移值和阈值比较
- 全部通过才返回 OK

### 第 166-223 行：`build_send_fields`

这是一个非常关键的函数。

它把当前已有的：

- 初始化参数
- 单次输入参数
- 图像预处理结果
- 分割结果

整理成最终的 `SendDataData`。

也就是说：

`sendData` 真正输出前，先在这里把字段全部算好。`

这里你要重点关注几类字段来源：

- 直接来自 `receive_fields`
  比如相机名、二维码、电压、电流
- 直接来自 `init_paras`
  比如 OH 上下限
- 由分割图代理计算出来
  比如层数、对齐度、OH 子结果
- 由多个子结果综合出来
  比如 `final_result_flag`

### 第 228-236 行：构造函数和析构函数

构造函数里做的事：

1. 调用 `Base()` 构造基类
2. 设置 Qt 的 parent
3. 在外部状态表里为当前对象创建一份 `AdapterState`

析构函数里做的事：

- 从状态表里删除当前对象对应的状态

### 第 238-266 行：`initParas`

这是正式生命周期第一步。

流程如下：

1. 从状态表拿到当前对象的状态
2. 解码 `config`
3. 读取模型路径
4. 检查模型文件是否存在
5. 初始化 ONNX Runtime
6. 设置状态标志位
7. 把路径同步回头文件成员变量

你可以把它理解成：

`把系统启动起来，准备好模型。`

### 第 268-304 行：`reciveData`

这是第二步。

流程如下：

1. 先确认已经做过 `initParas`
2. 检查是否真的传入了图像
3. 解码本次输入字节流
4. 调用 `prepare_input_images(...)`
5. 把预处理后的图同步到类成员变量
6. 设置状态标志

你可以把它理解成：

`把本次要检测的图像和附加参数装进系统。`

### 第 306-331 行：`run`

这是第三步，也是最核心的一步。

流程如下：

1. 确认已经 `init` + `receive`
2. 生成模型输入裁切图
3. 调用 `SegRuntime::run(...)` 做推理
4. 把局部结果恢复到整图
5. 生成 `m_segImage`
6. 生成 `m_segImageTo255`
7. 生成绘制图
8. 计算 `send_fields`
9. 设置 `has_run = true`

也就是说：

`所有真正的算法处理都集中在这一步。`

### 第 333-349 行：`sendData`

这一步只做输出，不再重新计算。

它做的事：

1. 检查 `run` 是否已经完成
2. 按 PDF 约定顺序把 5 张图放进 `images`
3. 把 `send_fields` 编码成 `QByteArray`

注意：

这 5 张图的顺序不能乱，因为外部接口就是按这个顺序约定的。

### 第 351-359 行：`encodeResultValue`

这是一个辅助接口。

当前实现里它没有真的使用传进来的 `append_value`，而是直接把当前状态里的 `send_fields` 重新编码。

所以你可以把它理解成：

`一个简化版的 sendData 数据编码入口`

### 第 361-365 行：`decodeResult`

这里很简单：

1. 把字节流按 UTF-8 变成字符串
2. 按换行拆开
3. 返回 `QVector<QString>`

本质上就是：

`把 sendData 的字节流反向拆成可读文本行`

### 第 367-375 行：导出接口

```cpp
extern "C" {
Q_DECL_EXPORT OH_Detect_Station_Module* create();
Q_DECL_EXPORT void destory(...);
}
```

这段的意义是：

- 给动态库导出一个统一的创建函数
- 给动态库导出一个统一的销毁函数

为什么要 `extern "C"`？

因为这样导出的函数名更稳定，方便别的程序按固定名字去加载。

## 12. 你最容易卡住的 8 个点

### 1. 为什么头文件和 cpp 分开？

因为 C++ 通常把“声明”和“实现”拆开，便于编译和组织工程。

### 2. 为什么 `OH_Detect_Station_Module.cpp` 里没有在类里直接加成员？

因为这里要尽量不改交付头文件的接口形态，所以用了外部 `state_map()` 来托管状态。

### 3. 为什么 `QByteArray` 不是 JSON？

因为当前 PDF 协议定义的是固定顺序字段，不是 JSON 协议。

### 4. 为什么 ONNX 输入要转成三通道？

因为当前模型输入是 `[1, 3, 768, 768]`。

### 5. 为什么还要 `seg_map` 和 `seg_map255` 两张图？

- `seg_map`
  更偏算法内部使用，保留类别值
- `seg_map255`
  更偏可视化使用，非零位置统一变 255

### 6. 为什么 `run()` 不直接输出，而是 `sendData()` 再输出？

因为接口生命周期就是这样设计的：

- `run()`
  负责计算
- `sendData()`
  负责取结果

### 7. 为什么这里一直在说“代理测量值”？

因为当前参考实现不是原始生产算法，只是用分割结果做近似测量，复现协议行为。

### 8. 为什么文档里一直强调 `best.onnx` 不能单独等同于完整交付模块？

因为模型文件只负责推理。

真正的完整模块还包括：

- 参数解码
- 图像预处理
- 输出解析
- 结果打包
- 5 张图生成

这些都在适配逻辑里，不在 ONNX 文件里。
