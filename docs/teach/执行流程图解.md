# `reference_qt_adapter` 执行流程图解

这份文档不重点讲语法，而是重点讲：

`外部调用这套 Qt 参考适配器时，程序到底按什么顺序跑下去。`

如果你把本文档和 [C++零基础语法入门.md](/D:/code/OH_Detect_Station_Module/docs/teach/C++零基础语法入门.md:1) 一起看，会更容易理解整套代码。

---

## 1. 先看总流程

外部使用者应该按这个顺序调用：

```text
create()
  ->
initParas(config)
  ->
reciveData(images, data)
  ->
run()
  ->
sendData(images, data)
  ->
destory(instance)
```

你可以把它理解成：

1. 创建对象
2. 初始化模型和配置
3. 送入一张待检测图像和本次输入参数
4. 执行推理和后处理
5. 取出结果图和结果字节流
6. 销毁对象

`delivery/reference_qt_adapter/` 里的代码可以分成 4 层：

简单说：

### 第三层：推理层

### 第四层：接口胶水层

## 3. 创建对象后，内部先做了什么

当外部调用 `create()` 创建模块对象时，最后会进入 [OH_Detect_Station_Module.cpp](D:/code/OH_Detect_Station_Module/delivery/reference_qt_adapter/OH_Detect_Station_Module.cpp:210) 附近的构造逻辑。

这里最重要的一件事是：

`给每个模块对象分配一份独立的 AdapterState。`

这个 `AdapterState` 里保存了：

- 初始化参数 `init_paras`
- 本次输入参数 `receive_fields`
- 最终输出参数 `send_fields`
- ONNX 推理器 `runtime`
- 预处理后的中间图
- 裁切信息
- 分割结果图
- 当前是否已经初始化、是否收到数据、是否已运行

为什么要单独放一份 `AdapterState`？

因为交付头文件本身没有为参考实现预留私有成员，所以只能在 `.cpp` 里额外托管状态。

你可以把它理解成：

`这个模块对象的“随身工作台”。`

## 4. `initParas(config)` 具体做什么

这一阶段的输入是一个 `QByteArray config`。

当前参考实现把它当成：

`按行组织的 UTF-8 文本字节流`

也就是说，它不是二进制结构体，也不是 JSON，而是类似这样：

```text
字段1
字段2
字段3
...
```

### 第一步：协议解码

[protocol_codec.cpp](D:/code/OH_Detect_Station_Module/delivery/reference_qt_adapter/protocol_codec.cpp:66) 附近的 `decode_init_paras` 会做这些事：

1. 把 `QByteArray` 按 UTF-8 转成文本
2. 按换行拆成多个字段
3. 检查字段数量是不是 36 个
4. 把每一行解析到 `InitParasData` 结构体

解析后的内容包括：

- ONNX 模型路径
- 图像保存路径
- 窗宽窗位
- OH 阈值
- 对齐阈值
- 层数检测目标
- 补偿量
- 若干开关

### 第二步：检查模型路径

解码成功后，`initParas` 会检查模型文件是否存在。

如果路径不存在，就直接失败。

### 第三步：初始化 ONNX Runtime

然后会调用 [seg_runtime.cpp](D:/code/OH_Detect_Station_Module/delivery/reference_qt_adapter/seg_runtime.cpp:17) 附近的 `SegRuntime::init`：

- 创建 `SessionOptions`
- 加载 ONNX 模型
- 读取输入名
- 读取所有输出名

这一阶段成功后，模块进入“已初始化”状态。

---

## 5. `reciveData(images, data)` 具体做什么

这一阶段有两类输入：

- `images`
  一组图像
- `data`
  本次检测的附加字段字节流

### 第一步：解码本次输入字段

[protocol_codec.cpp](D:/code/OH_Detect_Station_Module/delivery/reference_qt_adapter/protocol_codec.cpp:106) 附近的 `decode_receive_data` 会把 `data` 解码成 6 个字段：

- 相机名
- 产品二维码
- 管电压
- 管电流
- 管寿命
- 治具号

### 第二步：检查图像是否合规

然后 `reciveData` 会检查：

- 是否真的传了图像
- 第一张图是不是非空
- 类型是不是 `CV_16UC1`

如果不是 16 位单通道图，就失败。

### 第三步：做图像预处理

[image_pipeline.cpp](D:/code/OH_Detect_Station_Module/delivery/reference_qt_adapter/image_pipeline.cpp:39) 附近的 `prepare_input_images` 会做 3 件事：

1. 统一图像尺寸
2. 按窗宽窗位把 16 位图映射成 8 位图
3. 做 CLAHE 和高斯模糊增强

处理后会得到三张中间图：

- `standardized16`
  标准化后的 16 位图
- `raw8`
  映射后的 8 位图
- `enhanced8`
  增强后的 8 位图

这一阶段成功后，模块进入“已收到本次输入”状态。

---

## 6. `run()` 具体做什么

这是整套流程的核心。

可以把它理解成：

`真正开始干活。`

### 第一步：构造模型输入图

[image_pipeline.cpp](D:/code/OH_Detect_Station_Module/delivery/reference_qt_adapter/image_pipeline.cpp:61) 附近的 `build_model_input_bgr` 会：

1. 从增强后的灰度图里做中心裁切
2. 保证裁切结果是 `768 x 768`
3. 把灰度图转成 BGR 三通道图

为什么要转成三通道？

因为当前 ONNX 模型要求输入形状是：

`[1, 3, 768, 768]`

### 第二步：把 OpenCV 图像改写成 ONNX 输入张量

[seg_runtime.cpp](D:/code/OH_Detect_Station_Module/delivery/reference_qt_adapter/seg_runtime.cpp:218) 附近会把 `cv::Mat` 写进一个 `std::vector<float>`。

这里做的是：

- 像素值除以 `255.0`
- 从 OpenCV 常见的 HWC 排列改成 ONNX 需要的 CHW 排列

你可以先记一句：

`OpenCV 看图像，ONNX 看张量。`

所以中间一定要做一次格式转换。

### 第三步：执行 ONNX 推理

接着调用 `session_->Run(...)`，拿到模型输出。

当前模型输出有两个：

- 一个 3 维张量，代表 boxes/class/mask 系数
- 一个 4 维张量，代表 proto 掩码基底

### 第四步：解析 YOLOv8-seg 输出

[seg_runtime.cpp](D:/code/OH_Detect_Station_Module/delivery/reference_qt_adapter/seg_runtime.cpp:69) 附近的 `parse_detections` 会：

1. 判断哪个输出是 boxes，哪个是 proto
2. 算出类别数量
3. 遍历每个候选框
4. 选每个候选框分数最高的类别
5. 低于阈值的候选框直接丢掉
6. 保存 box、score、mask 系数

### 第五步：做 NMS

[seg_runtime.cpp](D:/code/OH_Detect_Station_Module/delivery/reference_qt_adapter/seg_runtime.cpp:138) 附近的 `apply_nms` 会把重叠太高的框去掉。

这样可以避免多个框重复指向同一目标。

### 第六步：重建分割图

[seg_runtime.cpp](D:/code/OH_Detect_Station_Module/delivery/reference_qt_adapter/seg_runtime.cpp:163) 附近的 `build_segmentation_maps` 会：

1. 用 mask 系数和 proto 做线性组合
2. 过 `sigmoid`
3. resize 回输入图大小
4. 在 box 范围内按阈值生成二值 mask
5. 汇总到两张结果图里

最终得到：

- `seg_map`
  类别分割图
- `seg_map255`
  255 二值掩码图

### 第七步：把裁切结果贴回整图

模型实际只看了裁切区域，所以 `run()` 后面还会把裁切区域内的结果恢复回整张图大小。

这一步由 [image_pipeline.cpp](D:/code/OH_Detect_Station_Module/delivery/reference_qt_adapter/image_pipeline.cpp:76) 的 `restore_crop_to_full_size` 完成。

## 7. `run()` 结束后程序里已经有哪些结果

到了这一步，程序内部已经有这些东西：

- 预处理后的图像
- 全尺寸分割图 `seg_map`
- 全尺寸二值图 `seg_map255`
- 一些可用于代理测量的几何信息

但这些结果还只是“内部中间结果”，还没有按交付协议整理好。

## 8. `sendData(images, data)` 具体做什么

这一阶段负责：

- 生成 5 张交付图片
- 生成最终结果字节流

### 第一步：生成 `send_fields`

[OH_Detect_Station_Module.cpp](D:/code/OH_Detect_Station_Module/delivery/reference_qt_adapter/OH_Detect_Station_Module.cpp:148) 附近的 `build_send_fields` 会根据：

- `init_paras`
- `receive_fields`
- `preprocessed`
- `seg_map`
- `seg_map255`

计算出最终的 `SendDataData`。

这里主要做了几类事情：

- 扫码结果判定
- 图像平均灰度统计
- OH 代理判定
- 对齐代理判定
- 层数代理判定
- 最终总结果判定

要特别注意：

这里的很多值是“参考适配器里的代理测量值”，不是原始生产算法的真实实现。

### 第二步：编码成字节流

[protocol_codec.cpp](D:/code/OH_Detect_Station_Module/delivery/reference_qt_adapter/protocol_codec.cpp:117) 的 `encode_send_data` 会把 `SendDataData` 按固定字段顺序重新拼成 UTF-8 `QByteArray`。

也就是说：

- 程序内部用结构体方便处理
- 对外输出时再按协议顺序拼回字节流

### 第三步：生成输出图片

`sendData` 还会把结果图整理到 `images` 输出数组里。

一般包括这类内容：

- 原始或标准化后的图
- 8 位增强图
- 分割结果图
- 绘制轮廓后的调试图
- 其他协议约定图像

其中绘制轮廓图会用到 [image_pipeline.cpp](D:/code/OH_Detect_Station_Module/delivery/reference_qt_adapter/image_pipeline.cpp:88) 的 `build_drawing_image`。

## 9. 为什么必须按顺序调用

这套代码内部维护了三个关键状态：

- `has_init`
- `has_receive`
- `has_run`

所以调用顺序不能乱。

例如：

- 没 `initParas` 就 `run`
  不行，因为模型都还没加载
- 没 `reciveData` 就 `run`
  不行，因为本次图像和输入字段还没准备好
- 没 `run` 就 `sendData`
  不行，因为结果还没算出来

所以它本质上是一个“分阶段推进”的状态机。

## 10. 用一句话概括每一步

如果你想快速记忆，可以只记下面这 4 句：

- `initParas`
  把“长期配置”装进来，并把模型加载好
- `reciveData`
  把“本次图像和本次字段”装进来，并做好预处理
- `run`
  真正执行模型推理和后处理
- `sendData`
  把内部结果整理成交付协议要的图片和字节流

## 11. 你可以怎样带着问题去读源码

建议你第一次读源码时，只追这 5 个问题：

1. 输入字节流在哪里被拆开
2. 图像在哪里从 16 位变成 8 位
3. 模型输入是在哪里变成 `[1, 3, 768, 768]`
4. 模型双输出是在哪里被重新解析成分割图
5. 最终 `sendData` 是在哪里被重新拼出来

对应文件就是：

- 问题 1：`protocol_codec.cpp`
- 问题 2：`image_pipeline.cpp`
- 问题 3：`image_pipeline.cpp` + `seg_runtime.cpp`
- 问题 4：`seg_runtime.cpp`
- 问题 5：`OH_Detect_Station_Module.cpp` + `protocol_codec.cpp`

## 12. 最后用最白话的方式再说一遍

你可以把这套参考适配器想成一条流水线：

```text
原始 16 位图 + 输入字节流
    ->
协议解码
    ->
图像预处理
    ->
ONNX 推理
    ->
分割结果恢复
    ->
代理测量与判定
    ->
输出图片 + sendData 字节流
```

你以后如果看代码看乱了，就回到这条线。

只要你知道当前这段代码处在流水线的哪一段，理解就不会完全丢掉。
