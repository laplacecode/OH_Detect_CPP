本文用于整理本项目从客户输入资料到当前交付输出的完整需求边界，明确客户真正需要的内容、客户已提供的信息、当前仓库已有能力、已经生成的交付包，以及后续仍需确认或优化的事项。

## 2. 客户给出的资料

### 2.1 Release 链接库要求

客户提到 Release 版本需要链接以下库：

```text
Hd_AI_Station_Jr_module.lib
Pack_AI_module.lib
opencv_world4100.lib
```

这说明客户原有接入方式不是单纯加载 ONNX，而是通过算法模块和打包模块完成集成。

### 2.2 DLL 动态库要求

客户要求：

```text
将 3 个文件夹内所有的 dll 拷贝到软件生成目录下
```

这里的软件生成目录应理解为客户最终 `.exe` 所在目录。运行时所有算法 DLL、Pack DLL、OpenCV DLL、Qt DLL、ONNX Runtime DLL 以及编译器运行库 DLL 都需要能被系统加载。

### 2.3 模型配置要求

客户要求：

```text
将模型 model 文件夹拷贝到程序生成目录下
```

目标目录结构应为：

```text
程序生成目录/
  model/
    best.onnx
```

如果客户现场仍使用旧模型文件名，例如：

```text
602_JiEr_Detect_EigghtClasses_1024.onnx
```

则需要把该文件放到 `model/` 下，并同步修改程序传入的模型路径。

### 2.4 客户旧版调用示例

客户给出的旧版调用流程大致如下：

```cpp
QString currPath = QCoreApplication::applicationDirPath();
QString mdName = "Hd_AI_Station_Jr_module";
QString modulePath = QDir(currPath).filePath(mdName);
QString onnxPath = QDir(currPath).filePath("model/best.onnx");
QString imgPath = "D:/jier_Right.tif";

cv::Mat image = cv::imread(imgPath.toStdString(), cv::IMREAD_UNCHANGED);

if (!image.empty()) {
    myLibrary* myLib = new myLibrary();

    myLib->loadModule(modulePath, mdName);
    myLib->initModule(onnxPath, 0.0235);
    myLib->receive(image, "123");
    myLib->run();

    bool result = myLib->send();
    qDebug() << "result:" << result;

    delete myLib;
}
```

从该示例可以看出客户原有链路包含：

```text
loadModule -> initModule -> receive -> run -> send
```

其中 `myLibrary` 属于客户侧或 Pack 层加载器，不属于本算法模块本身。

## 3. 客户真实需求拆解

客户实际需求可拆成两层。

### 3.1 模型验证需求

客户希望确认：

```text
ONNX 模型可以被加载
输入图像可以被读取并送入算法
推理可以执行
结果图或结果数据可以输出
```

这类需求可以通过一个独立验证包完成，不要求客户配置 C++ 编译环境。

### 3.2 软件集成需求

客户希望把算法接入原有软件，因此需要：

```text
算法 DLL
导入库 lib
头文件 h
模型 model
运行依赖 dll
接口说明文档
验证程序或示例调用方式
```

这类需求不仅需要 ONNX，还需要前处理、后处理、协议编码、结果输出和 DLL 导出接口。

## 4. ONNX 与算法 DLL 的关系

ONNX 文件不是完整算法 DLL。

```text
ONNX 模型 = 神经网络结构和权重
算法 DLL = ONNX 模型 + 图像前处理 + ONNX Runtime 推理 + 后处理 + 业务结果计算 + 接口封装
```

当前项目中的 `best.onnx` 是算法模块中的模型部分。要让客户软件调用，需要配套 C++ 代码和 DLL 封装。

## 9. 需要特别说明的兼容性问题

### 9.1 编译环境

当前 `Hd_AI_Station_Jr_module.dll/.lib` 基于本机 MinGW 环境生成。

如果客户工程也是 MinGW/Qt 体系，可以直接参考当前包。

如果客户要求 MSVC、Visual Studio 或固定的 `opencv_world4100.lib`，则需要使用客户指定的编译器和依赖版本重新构建一版交付包。

### 9.2 OpenCV 版本

客户给出的库名是：

```text
opencv_world4100.lib
```

当前本机发布包使用的是 MSYS2 MinGW OpenCV 4.13 的拆分 DLL，例如：

```text
libopencv_core-413.dll
libopencv_imgproc-413.dll
libopencv_imgcodecs-413.dll
```

因此，如果客户环境强制要求 OpenCV 4.10 world 库，需要后续做 OpenCV 4.10 对齐构建。

### 9.3 Pack_AI_module

客户资料中提到：

```text
Pack_AI_module.lib
```

当前仓库没有 `Pack_AI_module.dll/.lib` 源码或产物。当前验证包绕过 Pack 层，直接用 `verify_dll_smoke.exe` 动态加载 `Hd_AI_Station_Jr_module.dll` 的 `create/destory` 接口。

如果客户必须通过 `Pack_AI_module` 和 `myLibrary` 调用，则需要补充 Pack 层产物或确认客户现场已有该模块。

### 9.4 模型文件名

当前正式模型名为：

```text
model/best.onnx
```

客户旧资料中的模型名是：

```text
model/602_JiEr_Detect_EigghtClasses_1024.onnx
```

两者需要在交付前统一。若保持 `best.onnx`，客户代码中的模型路径应同步调整。

## 11. 后续优化建议

建议后续优化按优先级处理：

1. 对齐客户真实编译环境：确认 MinGW 还是 MSVC，确认 OpenCV 4.10 还是当前 4.13。
2. 对齐客户运行框架：确认客户是否必须使用 `Pack_AI_module` 和 `myLibrary`。
3. 精简运行依赖 DLL：当前包为了保证可运行，复制了较多 OpenCV DLL；后续可做依赖扫描和瘦身。
4. 固化发布脚本：继续维护 `package_release.ps1`，保证每次交付包可重复生成。
5. 补充客户集成示例工程：如客户需要自行编译，可提供最小 Qt/C++ 示例工程。
